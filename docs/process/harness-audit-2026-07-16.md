## Кратко

- Харнесс **выше среднего**: Scope — образцовый (5/5), документация актуальна, verification реально работает.
- **Bottleneck — Lifecycle (2/5):** ни начало, ни конец сессии не автоматизированы вообще; всё держится на дисциплине модели.
- Три конкретных дыры: `progress.md` разросся до 127 KB и продолжает читаться при старте; CI существует, но **не является required check** (merge проходит до завершения CI); качество AI-текста — ядро продукта — не имеет регрессионного прогона (#162 открыт).
- «Пишет ли модель код только под тесты?» — риск реален для промпт-тестов (spec-mirroring), но купирован **не тестами**, а живыми `eval:text`-прогонами + судьёй + проверкой глазами. Это правило нигде не записано — работает де-факто.

## Что происходит в конце сессии

**Обновления документации НЕ запускаются автоматически.** Session-end checklist в AGENTS.md (обновить `progress.md`, закрыть issues, прогнать `init.sh`, закоммитить) — полностью ручной. Нет ни Stop-hook, ни какой-либо иной автоматики.

Единственная автоматика во всём харнессе — husky pre-commit → prettier (`lint-staged`).

То, что в последних сессиях `progress.md` исправно обновлялся — заслуга дисциплины модели, а не харнесса. Это хрупко: другая сессия может просто забыть, и харнесс никак не напомнит.

## Запускаются ли тесты реально?

**Да, на трёх уровнях — с одной дырой:**

1. **`./init.sh` реально запускается и реально ловит проблемы.** В сессиях 2026-07 он находил настоящие FAIL'ы (устаревшие тесты промптов после изменения поведения, lint-ошибки), которые чинились до коммита.
2. **CI на GitHub реально гоняет `./init.sh` на каждый PR** — последние 5 PR: все проверки SUCCESS.
3. **Дыра: CI не является required status check** (проверено через `gh api .../branches/main/protection` — `required_status_checks` отсутствует). Агент self-merge'ит PR мгновенно, не дожидаясь CI. Локальный `init.sh` компенсирует, но это конвенция, а не гарантия: если модель однажды пропустит локальный прогон, красный код уйдёт в `main` без препятствий.

## Пишет ли модель код «только под тесты»?

Картина смешанная, риск купирован — но не тестами:

- **Схемные/валидаторные тесты** — настоящие поведенческие («схема отклоняет `text-focus` для age 6») — хорошие.
- **Промпт-тесты** — spec-mirroring: `expect(prompt).toContain('Расплата')`. Тест и код пишутся одной рукой из одной спеки — такой тест ловит *регрессию*, но не проверяет, что промпт *работает*. Здесь риск «кода под тест» реален.
- **Настоящая защита качества — не unit-тесты, а `eval:text`** (живые прогоны пайплайна) + судья (registerMatch) + проверка глазами в проде. Промпт-изменения 2026-07 проверялись живыми прогонами — именно они ловили реальные проблемы (пропадающий питомец #243, абстрактные заголовки #232), которые unit-тесты не поймали бы никогда.
- Фактический режим — чаще **test-after** (фикс → тест), чем декларированный TDD. Для фиксов это нормально; для фич по плану TDD соблюдается (план #196 написан строго test-first).

## Рекомендации (по приоритету)

1. **Ротация `progress.md`** — архивировать старые записи в `docs/process/progress-archive-2026-Q2.md`, оставить последние ~10 записей + строку «архив там-то». Плюс правка AGENTS.md: «читай последние 2–3 записи, не весь файл». Самый дешёвый и самый быстро окупающийся фикс.
2. **Сделать CI required check** на `main` (одна команда `gh api`). Закрывает единственную реальную дыру verification: merge до завершения CI.
3. **Session-end автоматика** — минимальный Stop-hook в `.claude/settings.json` проекта, печатающий чеклист («progress.md обновлён? init.sh зелёный? issue закрыт?»). Превращает конвенцию в напоминание, которое невозможно не увидеть.
4. **#162 batch-eval — главный пробел.** Качество текста (ядро продукта) защищено только ручными прогонами. Скрипт «прогнать 5–8 целей × оба возраста, вывести таблицу registerMatch» дал бы регрессионную защиту промптов.
5. **Закрепить де-факто правило в AGENTS.md:** «для изменений AI-пайплайна — минимум один живой `eval:text`-прогон до PR». Сейчас это происходит, но нигде не записано.
6. **Гигиена:** почистить 229 permissions в `.claude/settings.local.json` (масса одноразовых захардкоженных команд — шум); глобально — отключить неиспользуемые в этом проекте MCP (computer-use, Figma/Linear): это уменьшит стартовый контекст сильнее, чем любая правка проектных файлов.
